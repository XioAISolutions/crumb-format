# v1.2 open question — how should `refs` resolve?

**Status:** open, draft. v1.2 ships with `refs` as an advisory pointer only. How a consumer turns a ref into a concrete CRUMB is left to implementations for now.

This doc frames the decision so we can pick one (or a hybrid) for v1.3.

## The ref syntax is decoupled from resolution

The header shape is fixed:

```text
refs=mem-prefs-abc123, map-web-app-2026q2
```

What a ref _means_ — where a consumer looks to fetch that CRUMB — is the open question.

## Candidate schemes

### A. Bare id (filename-matches-id convention)

`mem-prefs-abc123` → look for `mem-prefs-abc123.crumb` in a known directory.

- **Pro:** trivial. No new machinery. Works with `git add *.crumb` directly.
- **Pro:** Matches `crumb new` default filename behavior.
- **Con:** No global uniqueness guarantee. A ref emitted from repo A can't be resolved from repo B.
- **Con:** Ambiguous when the same id exists in multiple directories.

### B. Content hash

`sha256:abc123...` → consumer fetches by hash from a content-addressable store.

- **Pro:** Global uniqueness. Immutable references.
- **Pro:** Git-friendly — every git repo is a content store.
- **Con:** No one can hand-write hashes. Breaks the "tiny, human-readable" rule in §1.
- **Con:** Needs a resolver layer every consumer must implement.

### C. URL

`refs=https://crumbs.example.com/mem-prefs-abc123.crumb`

- **Pro:** Standard, familiar. Works for public CRUMBs out of the box.
- **Pro:** `url=` header already exists as optional spec-link metadata.
- **Con:** Ties a ref to a specific host. Dead links rot the graph.
- **Con:** Copy-paste to an air-gapped environment breaks.

### D. Registry lookup

A well-known namespace (`crumb.xiosolutions.com/...`) resolves to a canonical URL via a lookup service.

- **Pro:** Decouples identity from location.
- **Pro:** Allows revocation and versioning.
- **Con:** Adds a service dependency. CRUMBs are meant to travel freely.
- **Con:** Over-engineered for v1.x.

### E. Hybrid (recommended shape for v1.3)

Allow all four. Resolution order is consumer-specific, but the default SHOULD be:

1. Bare id → local directory (scheme A)
2. URL → fetch if network allowed (scheme C)
3. `sha256:` prefix → content store (scheme B)
4. Registry lookup only if explicitly enabled (scheme D)

A consumer that resolves nothing MUST NOT fail — refs are advisory.

## Scope questions

Two orthogonal questions that need answers alongside the scheme:

- **Cycle handling:** depth-limited walk with a visited-set. Limit default: 5. _Tentatively agreed in §9.3._
- **Missing refs:** warn or silent? `crumb lint` SHOULD warn; parsers MUST NOT fail. _Tentatively agreed in §9.1._
