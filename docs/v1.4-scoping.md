# CRUMB v1.4 — scoping draft

**Status:** Scoping only. Nothing on this page is normative. The purpose is to collect candidate additions and decide which deserve to ship before the next wire-format bump.

v1.3 landed `kind=agent`, `[handoff]` deps, `[workflow]`, `[checks]`, `[guardrails]`, `[capabilities]`, `[script]`, structured `[constraints]`, normative ref resolution (§17), and normative fold selection (§18). This doc frames the next round.

## Guiding constraints (unchanged)

- Hand-writable. Copy-paste friendly. One mobile screen per CRUMB.
- Purely additive wire-format bumps. A v1.3 parser must still accept a v1.4 file.
- CRUMB is a context format, not an orchestration runtime. Declarative over executable.

## Candidates

### 1. `kind=review` — formal PR review crumbs

**Pain:** Review comments scatter across PR UIs. A single handoff-shaped record of "what was reviewed, by whom, with what verdict" would let review state travel across tools.

**Sketch:**

```text
kind=review
---
[identity]
reviewer=alice
pr=42

[verdict]
status=changes-requested

[findings]
- path=auth.ts line=87  severity=blocker  note=…
- path=auth.ts line=104 severity=nit      note=…

[checks]
- ci :: pass
- tests :: pass
```

**Overlap:** Close to `kind=audit` (which already exists in `cli/crumb.py` REQUIRED_SECTIONS but isn't documented). Decide whether to promote `audit` to the spec or replace it with `review`.

**Open:** Does this serve GitHub/GitLab reviewers, CRUMB-native reviewers, or both? If both, a format-level spec may be premature — start with a `review/*` section convention on `kind=task`.

### 2. Typed `[checks]` thresholds as first-class

**Pain:** v1.3 `[checks]` currently uses free-form trailing `key=value`. Consumers have to parse `threshold=85` by string match. Formalizing types (numeric threshold, time-bounded, regex-match) would make the section machine-first without losing prose.

**Sketch:**

```text
[checks]
- coverage :: 87%  @type=percent  threshold=85  direction=min
- latency_p99 :: 42ms  @type=duration  threshold=100ms  direction=max
- version :: 1.2.3  @type=semver  range=>=1.0.0,<2.0.0
```

**Open:** `@type` per line is novel for CRUMB; v1.2 `@type:` only applies as the first line of a whole section. Precedent risk.

### 3. Ref resolver caching

**Pain:** v1.3 §17 resolution re-walks the ref graph on every parse. For a task crumb that refs a large map crumb, that's expensive.

**Sketch:** A SHA-256-addressed on-disk cache (`$CRUMB_HOME/cache/<digest>/`) that short-circuits the bare-id → local-dir step.

**Non-goal:** Adding caching is an implementation detail, not a spec change. Belongs in `cli/ref_resolver.py`, not in SPEC.md. Listed here only so it doesn't get forgotten.

### 4. Bench scoring overhaul

**Pain:** `crumb bench` shipped at v0.2 and hasn't been revisited. It rewards density and compressibility but doesn't know about folds, priorities, or delta crumbs. A v1.3-aware bench would:

- Surface `@priority` distribution.
- Flag folds with summary/full token ratios outside a sensible band (e.g. summary > 50% of full = low compression value).
- Credit `refs=sha256:…` usage (content-addressed refs = sender expects receiver to elide).

**Non-goal:** Not a spec change either — implementation-only. But worth coordinating with the bench comment in `.github/workflows/bench-pr.yml` so the PR comment surface is coherent.

### 5. `kind=receipt` — after-the-fact verification record

**Pain:** `[checks]` records what the sender verified *before* handoff. There's no matching shape for the receiver to record what they verified *after* completing the work.

**Sketch:**

```text
kind=receipt
---
[identity]
completed_by=claude-agent
for_task=sha256:abc123…

[delivered]
- fix.ts written
- tests pass

[residual]
- remaining questions for reviewer
```

**Open:** Is this worth a new kind, or is it `kind=delta` plus a convention?

### 6. `[handoff]` `until=` deadlines

**Pain:** `[handoff]` lines accept an optional `deadline=` (SPEC §11.1) but the format isn't normative. No parser enforces a shape. v1.4 could pick ISO-8601 normatively and let `crumb lint` warn on malformed deadlines.

**Cost:** Minimal. Single-line addition to §11.1.

## Out of scope (explicit)

- Anything embedding executable behavior beyond v1.3 `[script]`'s opaque carrier.
- Cross-agent coordination (consensus, voting, routing) — different layer.
- Encryption/signing — use the transport, not the format.

## Process

Rough cut of priority for the v1.4 cycle:

1. **Ship**: #6 (deadlines normative) — trivial.
2. **Prototype**: #2 (typed checks), #3 (resolver cache), #4 (bench overhaul).
3. **Discuss**: #1 (kind=review vs kind=audit), #5 (kind=receipt).

Timeline: no commitment. Open a scoping issue when each candidate is ready to move.

## Closing

This doc exists so future CRUMB maintainers have a ledger to scan when they're deciding what to build next. Feel free to add entries, cross items off, or split this file into per-candidate docs under `docs/v1.4/`.
