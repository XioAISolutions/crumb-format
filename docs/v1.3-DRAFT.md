# CRUMB v1.3 — draft spec

**Status:** draft for review.
**Target version header:** `v=1.3`.
**Compatibility posture:** purely additive. A v1.2 parser accepts a v1.3 file by ignoring unknown headers and sections (SPEC §8). A v1.3 parser accepts `v ∈ {1.1, 1.2, 1.3}`.

Scope:

1. **Close** the two design questions v1.2 left open (`refs` resolution, fold heuristic).
2. **Extend** the existing `[handoff]` primitive with ordering dependencies.
3. **Formalize** structured lines in `[constraints]`.
4. **Add** optional sections: `[checks]`, `[guardrails]`, `[capabilities]`, `[script]`, `[workflow]`, and `[invariants]` on `kind=task`.
5. **Add** a new kind: `kind=agent` for reusable agent personas.

All additions are purely optional. A parser that ignores every v1.3 addition still conforms.

---

## 1. Closing v1.2 open question: `refs` resolution

v1.2 SPEC §9 ships `refs` as an advisory pointer with resolution left to implementations. v1.3 normatively specifies the default.

### 1.1 Normative resolution order

A conforming consumer MUST attempt resolution in this order and stop at the first hit:

1. **Bare id → local directory.** Look for `<id>.crumb` in a consumer-configured search path. Default search path: current working directory, then `$CRUMB_HOME` (default `~/.crumb/`).
2. **`sha256:<hex>` digest → content store.** If a consumer maintains a content-addressed store (`$CRUMB_STORE`, default `~/.crumb/store/`), look up by digest.
3. **URL → fetch.** Only if network is explicitly enabled by the consumer. Default: disabled.
4. **Registry lookup.** Only if a registry is explicitly configured. Default: none.

A consumer MUST NOT fail a parse on an unresolvable ref. Refs are advisory.

### 1.2 Cycle handling

Consumers MUST walk refs with a visited-set. Default depth limit: **5**. Consumers MAY configure.

### 1.3 `crumb lint` contract

`crumb lint` SHOULD warn on unresolvable refs when run with `--check-refs`. Parsers MUST NOT warn.

## 2. Closing v1.2 open question: fold heuristic

v1.2 SPEC §10 ships fold pairs with consumer selection left open. v1.3 normatively specifies the default.

### 2.1 Default selection: size-greedy with summary floor

A conforming budget-aware consumer MUST:

1. First, count every `[fold:X/summary]` as mandatory.
2. For each fold, attempt to upgrade `/summary` to `/full` in **declaration order** until the remaining token budget would be exceeded.
3. Never load both `/summary` and `/full` for the same NAME.

### 2.2 Writer override

A writer MAY declare `fold_priority=` as an optional header listing fold NAMEs in upgrade order:

```text
fold_priority=context, constraints
```

When present, consumers MUST honor this order over declaration order.

### 2.3 Both variants loaded simultaneously

Forbidden. A consumer that loads both is non-conforming.

## 3. `[handoff]` dependencies

v1.3 adds optional explicit dependencies so non-linear handoff graphs can be expressed without inventing a `[workflow]` section for simple cases. (`[workflow]`, §9, covers state-machine use cases.)

### 3.1 Syntax

Two new optional keys on a `[handoff]` line:

- `id=<token>` — stable identifier. Tokens are `[a-zA-Z0-9_-]+`.
- `after=<token>[,<token>...]` — comma-separated list of ids this step depends on.

```text
[handoff]
- id=repro   to=any    do=reproduce the failing test
- id=fix     to=any    do=propose a fix                  after=repro
- id=test    to=any    do=add regression test            after=fix
- id=review  to=human  do=approve before merge           after=test
```

### 3.2 Execution semantics

- A step with unmet `after=` dependencies is **blocked** and MUST NOT run.
- Earlier-line priority (v1.2 §11.2) still applies as a tiebreaker among ready steps.
- A `- [x]` completed line satisfies dependencies on its id.
- Cycles MUST be detected. Consumers SHOULD warn and fall back to linear order.

### 3.3 Backward compatibility

A v1.2 consumer that sees `id=` / `after=` as unknown keys treats the whole line as a bullet. Linear execution still works.

## 4. Structured `[constraints]` lines

v1.1+ `[constraints]` bodies are free prose. v1.3 formalizes an **optional** structured bullet form. Mixed bodies (prose + structured) are valid.

### 4.1 Syntax

```text
[constraints]
- Do not change the login UI                   # prose, unchanged
- deny=filesystem.write(/etc/**)               # structured
- require=regression_test_before_merge  why=audit
- prefer=incremental_patches
```

Recognized keys (all optional, all advisory):

- `deny=<expr>` — an action the receiving agent SHOULD NOT take.
- `require=<expr>` — a precondition the receiving agent SHOULD satisfy.
- `prefer=<expr>` — a soft preference.
- `why=<text>` — rationale.

Unknown keys do not invalidate a line.

---

## 5. `[checks]` section

Optional. Reports verification results at handoff time.

### 5.1 Syntax

```text
[checks]
- tests.test_auth.py :: pass
- coverage :: 87%          threshold=85
- lint :: fail             note=unused import in auth.py:12
- type_check :: pass
```

Grammar:

```text
check_line := "- " name " :: " status (" " key "=" value)*
status     := "pass" | "fail" | "skip" | free-form token
```

Unknown statuses and unknown trailing keys MUST NOT break parsing.

---

## 6. `[guardrails]` section

Optional. A structured list of enforcement hints for a receiving runtime or policy engine. CRUMB parsers do not enforce; `[guardrails]` is a travel-safe carrier for policy that external tools (AgentAuth, MCP policy engines, CI guards) can read.

### 6.1 Syntax

One guardrail per line. Recognized keys (all optional, all advisory):

- `type=<tool|scope|verify|approval>` — what kind of guardrail this is.
- `deny=<expr>`, `require=<expr>`, `max=<expr>`, `min=<expr>`, `who=<expr>`, `action=<expr>` — operands specific to the type.
- `why=<text>` — rationale.

```text
[guardrails]
- type=tool      deny=shell-exec      why=security boundary
- type=scope     max=files=5          why=token budget
- type=verify    require=tests        why=regression prevention
- type=approval  action=merge         who=human
```

### 6.2 Relationship to AgentAuth

A `[guardrails]` line is a hint. Actual enforcement is the consumer's job. When a CRUMB is received by an AgentAuth-aware runtime, the runtime SHOULD translate applicable lines into its policy engine (`crumb policy set`). A runtime without AgentAuth MAY ignore them — lines MUST NOT break parsing.

### 6.3 Relationship to `[constraints]`

`[constraints]` is human-authored prose + optional structure (§4). `[guardrails]` is machine-first structured hints. A writer MAY emit both; a reader SHOULD treat `[guardrails]` entries as denser, tool-readable expressions of the same intent.

## 7. `[capabilities]` section

Optional. Self-description carried with a handoff so the receiver knows what the sender could or could not do.

### 7.1 Syntax

Bullet lines with one or more of:

- `can=<expr>[,<expr>...]` — what the sender can do.
- `cannot=<expr>[,<expr>...]` — what the sender cannot do.
- `prefers=<expr>[,<expr>...]` — stylistic or architectural preferences.

```text
[capabilities]
- can=read_files, run_tests, git_commit
- cannot=deploy_prod, shell_exec
- prefers=incremental_patches
```

### 7.2 Relationship to AgentAuth passports

A passport is the authoritative source. `[capabilities]` is a transient, handoff-time snapshot. When both are present, the passport wins. When only `[capabilities]` is present, the receiver MAY use it as a soft hint.

## 8. `[script]` section

Optional. Carries an executable-intent block in a declared language. CRUMB parsers do not execute scripts; the section is an opaque body with a typed annotation that downstream runtimes may act on.

### 8.1 Syntax

The section body MUST begin with `@type: <lang>` and MAY include a `---` fence followed by the script body. Recognized `lang` values are advisory; unknown languages MUST NOT break parsing.

```text
[script]
@type: weave
@action: validate
---
agent.can("shell-exec") -> false
agent.must("run_tests") -> true
```

Suggested lang values: `weave`, `python`, `shell`, `javascript`, `json`.

### 8.2 Execution rules

- CRUMB parsers MUST NOT execute `[script]` bodies.
- A runtime that recognizes the `@type:` value MAY execute the body, subject to its own policy.
- `[script]` is a handoff primitive, not a program. Keeping actual code in a repo and referencing it via `[context]` is still the preferred shape. Use `[script]` only when the executable intent must travel with the handoff.

### 8.3 Size guidance

`crumb lint` SHOULD warn on `[script]` bodies larger than 2 KB. Large scripts belong in files, not in CRUMBs.

## 9. `[workflow]` section

Optional. Multi-step state machine for orchestration use cases that outgrow a linear `[handoff]` block.

### 9.1 Syntax

Numbered steps, one per line. Each line MAY carry:

- `id=<token>` — stable identifier. If omitted, the numeric prefix is the id.
- `status=<pending|blocked|in_progress|done|skipped>`.
- `owner=<expr>` — who performs this step (agent id, `human`, `any`).
- `depends_on=<id>[,<id>...]` — step dependencies.
- `why=<text>` — rationale.

```text
[workflow]
1. reproduce_bug     status=pending     owner=any
2. write_test        status=blocked     owner=any      depends_on=1
3. implement_fix     status=blocked     owner=any      depends_on=2
4. human_approval    status=blocked     owner=human    depends_on=3
```

### 9.2 Execution semantics

Same dependency resolution as `[handoff]` §3.2 — unmet `depends_on` blocks the step. Cycles SHOULD be detected and warned.

### 9.3 When to use `[workflow]` vs `[handoff]`

- Use `[handoff]` for linear or lightly-branched next-step instructions (3–5 steps).
- Use `[workflow]` when status tracking across multiple sessions matters, or when steps have non-trivial ownership (multiple agents, human approval gates).
- A CRUMB MAY contain both: `[handoff]` for the immediate next move, `[workflow]` for the long-running plan.

## 10. `[invariants]` on `kind=task`

`[invariants]` already exists on `kind=map` (SPEC §4.3) meaning architectural truths. v1.3 extends it to `kind=task` as an **optional** section capturing runtime assertions the receiving agent must maintain while performing the task.

### 10.1 Syntax

Free-form bullets, optionally structured:

```text
[invariants]
- auth.middleware completes before auth.state read
- tests.test_auth.py passes
- no new dependencies introduced
```

### 10.2 Relationship to `[constraints]` and `[checks]`

- `[constraints]` — what the receiver must not do.
- `[invariants]` — what must remain true throughout the work.
- `[checks]` — what was verified before handoff.

Overlap is permitted. Writers pick the section that reads most naturally.

## 11. `kind=agent`

A new kind for reusable agent personas. A `kind=agent` crumb describes who an agent is — identity, style, rules of engagement — without tying it to a specific task.

### 11.1 Required sections

- `[identity]` — role, style, scope.

### 11.2 Optional sections

- `[rules]` — bullet list of standing orders.
- `[knowledge]` — expertise and learning areas.
- `[capabilities]` — same shape as §7.
- `[guardrails]` — same shape as §6.

### 11.3 Example

```text
BEGIN CRUMB
v=1.3
kind=agent
id=code-reviewer-v2
source=human.notes
---
[identity]
role=senior_reviewer
style=focus_on_edge_cases

[rules]
- never approve without tests
- suggest perf optimizations when O(n^2) detected

[knowledge]
- expert=python,typescript
- learning=rust
END CRUMB
```

### 11.4 Usage

A task crumb references an agent crumb via `refs=<agent-id>`. The receiving runtime loads the agent crumb first to establish persona, then processes the task.

## 15. Non-goals

- No breaking changes to v1.1 or v1.2 files.
- No parser-side enforcement of `[guardrails]`, `[script]`, or `[workflow]` — those are declarative, consumed by runtimes.
- No required runtime dependency on AgentAuth.
