# CRUMB v1.3 — draft spec

**Status:** draft for review. Targets the next wire-format bump.
**Target version header:** `v=1.3`.
**Compatibility posture:** purely additive. A v1.2 parser accepts a v1.3 file by ignoring unknown headers and sections (SPEC §8). A v1.3 parser accepts `v ∈ {1.1, 1.2, 1.3}`.

Scope is deliberately tight. v1.3 exists to:

1. **Close** the two design questions v1.2 left open (`refs` resolution, fold heuristic).
2. **Extend** the existing `[handoff]` primitive with ordering dependencies.
3. **Formalize** structured lines in `[constraints]` without breaking prose.
4. **Add** one new optional section (`[checks]`) where no existing section fits.

Everything else proposed for v1.3 (active enforcement guardrails, embedded scripts, agent identity crumbs, `[workflow]` state machines) is **explicitly rejected** and documented in §7 below. The line v1.3 defends: CRUMB is a static description of state and intent. It is not a runtime, not an agent language, and not an identity store.

---

## 1. Closing v1.2 open question: `refs` resolution

v1.2 SPEC §9 ships `refs` as an advisory pointer with resolution left to implementations. v1.3 normatively specifies the default.

### 1.1 Normative resolution order

A conforming consumer MUST attempt resolution in this order and stop at the first hit:

1. **Bare id → local directory.** Look for `<id>.crumb` in a consumer-configured search path. Default search path: current working directory, then `$CRUMB_HOME` (default `~/.crumb/`).
2. **`sha256:<hex>` digest → content store.** If a consumer maintains a content-addressed store (`$CRUMB_STORE`, default `~/.crumb/store/`), look up by digest.
3. **URL → fetch.** Only if network is explicitly enabled by the consumer. Default: disabled.
4. **Registry lookup.** Only if a registry is explicitly configured. Default: none.

A consumer MUST NOT fail a parse on an unresolvable ref. Refs are advisory. This preserves v1.2 §9.1 semantics.

### 1.2 Cycle handling

v1.2 §9.3 is made normative: consumers MUST walk refs with a visited-set. Default depth limit: **5**. Consumers MAY configure.

### 1.3 `crumb lint` contract

`crumb lint` SHOULD warn on unresolvable refs when run with `--check-refs`. Parsers MUST NOT warn.

## 2. Closing v1.2 open question: fold heuristic

v1.2 SPEC §10 ships fold pairs with consumer selection left open. v1.3 normatively specifies the default.

### 2.1 Default selection: size-greedy with summary floor

A conforming budget-aware consumer MUST:

1. First, count every `[fold:X/summary]` as mandatory (same budget class as a required section).
2. For each fold, attempt to upgrade `/summary` to `/full` in **declaration order** until the remaining token budget would be exceeded.
3. Never load both `/summary` and `/full` for the same NAME.

Non-budget-aware consumers MAY load `/full` always or `/summary` always; they SHOULD document which.

### 2.2 Writer override

A writer MAY declare `fold_priority=` as an optional header, listing fold NAMEs in upgrade order:

```text
fold_priority=context, constraints
```

When present, consumers MUST honor this order over declaration order.

### 2.3 Both variants loaded simultaneously

Forbidden. A fold pair collapses into "two sections with a naming convention" if both load. Consumers that ignore this rule are non-conforming.

## 3. `[handoff]` dependencies

v1.2 §11 defines top-down ordering. v1.3 adds optional explicit dependencies so non-linear handoff graphs can be expressed without inventing a `[workflow]` section.

### 3.1 Syntax

Two new optional keys on a `[handoff]` line:

- `id=<token>` — stable identifier for the step. Tokens are `[a-zA-Z0-9_-]+`. If omitted, the 1-indexed line position (counting only non-completed lines) is the implicit id.
- `after=<token>[,<token>...]` — comma-separated list of ids this step depends on.

```text
[handoff]
- id=repro   to=any    do=reproduce the failing test
- id=fix     to=any    do=propose a fix without landing   after=repro
- id=test    to=any    do=add regression test             after=fix
- id=review  to=human  do=approve before merge            after=test
```

### 3.2 Execution semantics

- A step with unmet `after=` dependencies is **blocked** and MUST NOT run.
- Earlier-line priority (v1.2 §11.2) still applies as a tiebreaker among ready steps.
- A `- [x]` completed line satisfies dependencies on its id.
- Cycles MUST be detected. Consumers that detect a cycle SHOULD warn and fall back to linear order.

### 3.3 Backward compatibility

A v1.2 consumer that sees `id=` / `after=` as unknown keys treats the whole line as a bullet (v1.2 §11.1 already specifies this). Linear execution still works; dependency enforcement just degrades to "ordered."

## 4. Structured `[constraints]` lines

v1.1+ `[constraints]` bodies are free prose. v1.3 formalizes an **optional** structured bullet form that tools can parse. Mixed bodies (prose + structured) are valid.

### 4.1 Syntax

A constraint line MAY be:

```text
- Do not change the login UI                   # prose, unchanged
- deny=filesystem.write(/etc/**)               # structured
- require=regression_test_before_merge  why=audit  # structured w/ why
- prefer=incremental_patches
```

Recognized keys (all optional, all advisory):

- `deny=<expr>` — an action the receiving agent SHOULD NOT take. `<expr>` is opaque to the parser.
- `require=<expr>` — a precondition the receiving agent SHOULD satisfy.
- `prefer=<expr>` — a soft preference.
- `why=<text>` — rationale, mirroring `[handoff]` `why=`.

Unknown keys do not invalidate a line. Parsers that don't recognize structured form treat the whole line as a bullet.

### 4.2 Why not a new `[guardrails]` section

Because:

- CRUMB cannot enforce anything. It is a text format. Enforcement is AgentAuth's job (`crumb policy set`, `crumb passport register`).
- `[constraints]` already exists and means "things the next agent should respect." Splitting it into prose-vs-enforcement creates two sections for one concept.
- A consumer that wants to feed CRUMB constraints into a real policy engine can grep the `deny=` / `require=` lines. That is enough.

CRUMB carries constraints. A runtime enforces them. The spec should not blur the boundary.

## 5. `[checks]` section

An **optional** section for reporting verification results at handoff time. Scope: "what did the sender verify before handing off?"

### 5.1 Applicability

Optional on any `kind`. Most natural on `kind=task`.

### 5.2 Syntax

One check per line. Format:

```text
[checks]
- tests.test_auth.py :: pass
- coverage :: 87%          threshold=85
- lint :: fail             note=unused import in auth.py:12
- type_check :: pass
```

Grammar, line-by-line:

```text
check_line := "- " name " :: " status (" " key "=" value)*
status     := "pass" | "fail" | "skip" | free-form token
```

Unknown statuses and unknown trailing keys MUST NOT break parsing.

### 5.3 Why this is not duplicating `[handoff]` `- [x]`

`[handoff]` completion signal (`- [x] reproduced the bug on main@da5e312`) marks a **handoff step** as done. `[checks]` reports a **verification result** that is not itself a step. The two overlap on a single case ("I ran the tests") but diverge on the rest ("coverage is 87%" is not a handoff step).

If a writer wants, they can skip `[checks]` and put verification inside `[handoff]` as completed bullets. The section is additive — not loading it is always valid.

## 7. Rejected proposals (explicit)

This section exists so future proposers know what was considered and why it was dropped. Kept short; each item is its own paragraph.

### 7.1 `[guardrails]` as an enforcement surface

Rejected. CRUMB is a text format. A parser cannot "deny shell-exec" — only a runtime can. This repo already ships AgentAuth (`crumb passport`, `crumb policy`) for that purpose. A `[guardrails]` section pretending to enforce creates a false contract: the writer thinks the denial is honored; the reader has no way to guarantee it. The genuine need — carrying machine-readable denials across handoffs — is served by §4's structured `[constraints]`.

### 7.2 `kind=agent` identity crumbs

Rejected. Agent identity is AgentAuth's job. A reusable persona ("you are a senior reviewer") fits `kind=mem` with an `[identity]` subsection (which `kind=wake` already uses). Two systems for one concept confuses consumers about which to trust.

### 7.3 `[capabilities]` section

Rejected for the same reason as 7.2. A sender's capabilities travel in their passport, referenceable via `refs=`. Inlining them per-crumb is duplication, and stale capability lists are worse than none.

### 7.4 `[script]` with embedded DSL

Rejected with prejudice. An embedded Weave (or any other) execution block violates the Standalone Posture declared in v0.4.0 and the "hand-writable, one-mobile-screen" goal in §1 of the core SPEC. If a user wants to ship executable intent, they ship a script file and reference it from `[context]`. CRUMB does not embed other languages.

### 7.5 `[workflow]` as a state machine

Rejected as a new section. The genuine need (non-linear step dependencies) is served by §3's `after=` extension to `[handoff]`. Status fields (`pending` / `blocked` / `done`) partially overlap with `- [ ]` / `- [x]` and fully overlap with the computed dependency state. A separate section would fragment the handoff surface.

### 7.6 `[invariants]` on `kind=task`

Rejected as a cross-kind expansion. `[invariants]` already exists on `kind=map` (SPEC §4.3) meaning architectural truths. Re-purposing it on `kind=task` for runtime assertions overloads the name. The proposed examples ("tests pass", "no new dependencies") are verification results (→ §5 `[checks]`) or constraints (→ §4), not invariants.

## 9. Non-goals

Explicit for scope discipline:

- No new `kind`. The five stay.
- No enforcement semantics in the parser. Parsers stay advisory.
- No embedded scripting of any kind.
- No runtime dependency on AgentAuth. The two systems interoperate via `refs=` but neither requires the other.
- No breaking changes to v1.1 or v1.2 files. A v1.3 parser reads them unchanged.
