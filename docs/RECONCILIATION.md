## 5. Architectural decisions the reconciliation needs to answer

1. **Entry point:** `cli.crumb:main` (main) vs `crumb_cli:main` (r0.3)?
2. **Package layout:** multi-package (`cli`, `agentauth`, `mcp`, `api`, `a2a`) vs single-package (`cli`)?
3. **Framing:** "second-brain for AI" (main) vs "open context interchange protocol" (r0.3)?
4. **`bridge` semantics:** format adapter (main) vs MemPalace adapter (r0.3) — or both, nested?
5. **Version target:** `0.3.0` (matches r0.3's prep) or a new `0.3.0-rc1` for the merged surface?
6. **Golden-fixture conformance testing** (r0.3 only) — keep? If yes, main's examples need `.expected.json` siblings.
7. **Local-AI integration** (r0.3's `cli/local_ai.py`) — ship in 0.3 or defer?
8. **AgentAuth, A2A, Palace, Reflect** (main only) — these are *huge*; confirm they're wanted in 0.3 vs defer to 0.4.

## 6. Recommended path

**Option A — full merge into main (recommended).** Take main as the base (it has more code and tests), land r0.3's work on top as distinct commits:

1. Merge by category, not by commit:
   - Spec + validator merges first (low risk, additive)
   - Homebrew formula, browser extension, prompts, fixtures next (additive files)
   - `cli/pack.py`, `cli/linting.py`, `cli/extensions.py`, `cli/local_ai.py` next (new files, safe)
   - `cli/bridge.py` **renamed** to `cli/mempalace_bridge.py` + wired under main's `bridge` subparser as `bridge mempalace …` (resolves naming collision)
   - Infra merges: `publish-pypi.yml` env-token, `auto-crumb-pr.yml`, `scripts/bootstrap_update.py`
   - `api/server.py` — add r0.3's pack/bridge/lint endpoints to main's server
   - `mcp/server.py` — union TOOLS list
   - `cli/crumb.py` — port r0.3's subparser block for pack/bridge/lint
   - `pyproject.toml` — bump to 0.3.0, keep main's multi-package layout
   - `CHANGELOG.md` / `README.md` / `RELEASE_NOTES_0.3.0.md` — authored last, reflecting the combined surface
2. Each category becomes 1 commit. ~10 commits total.
3. Full test suite must pass at each step. Target: 294 (main) + new r0.3 tests all green.

**Option B — rebase r0.3 onto main.** Simpler in theory but messier in practice because r0.3's entry-point and packages restructuring conflicts at the pyproject.toml root. Not recommended.

**Option C — cherry-pick selected r0.3 commits.** Good if we want to defer some of r0.3's work (e.g., local_ai, Homebrew) to 0.4. More surgical; requires per-commit judgment.

## 7. Risk areas

- **AgentAuth + A2A + Palace + Reflect** (main, ~5k lines) are brand-new subsystems r0.3 hasn't seen. Integration tests pass in isolation; combined behavior with r0.3's pack/bridge is unverified.
- **Duplicate REST APIs** — main and r0.3 both built `api/server.py` from scratch. Their OpenAPI specs disagree on title/version/paths. Merge requires choosing one skeleton and porting the other's handlers in.
- **Entry-point change** breaks anyone with a shell alias or CI step that imports from `cli.crumb` directly vs `crumb_cli`.
- **Golden-fixture tests** (r0.3) will fail on main's examples until `.expected.json` siblings are produced for each.
- **Homebrew formula** pinned to a specific release — needs a v0.3.0 tag + tarball hash update after merge.
