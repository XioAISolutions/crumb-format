# v1.2 open question — which `[fold:X/...]` variant should a consumer load?

**Status:** open, draft. v1.2 ships with fold pairs (`/summary` and `/full`) validated and substitutable for a plain required section. How a consumer picks between variants under token pressure is left to implementations for now.

This doc frames the decision so we can pick one (or a hybrid) for v1.3.

## The fold shape is fixed

Writers emit pairs:

```text
[fold:context/summary]
short form

[fold:context/full]
long form
```

Validators enforce pairing (a `/full` without a `/summary` is an error). Consumers choose at load time.

## Candidate heuristics

### A. Declaration order (first wins)

If `/summary` appears before `/full` in the file, consumers load `/summary`. If `/full` appears first, consumers load `/full`.

- **Pro:** Zero-cost. Authors control preference by ordering.
- **Con:** Ordering is a hidden protocol. Future formatters might reorder sections and break this.
- **Con:** No token-budget awareness — order-first is a blunt instrument.

### B. Size-greedy (fit as much as possible)

Consumer sums projected tokens. Load `/full` where it fits, fall back to `/summary` otherwise.

- **Pro:** Deterministic. Maximizes information given a budget.
- **Pro:** Matches existing `max_index_tokens` / `max_total_tokens` semantics.
- **Con:** Starvation: a single oversized section can crowd out smaller ones. Need a per-section fairness pass.
- **Con:** "Tokens" is tokenizer-specific. Our `estimate_tokens()` is a ~4 char/token approximation.

### C. Emitter-weighted

Writer declares priority in a new header: `fold_priority=context, constraints, goal`.

- **Pro:** Writer knows which sections matter most for the handoff.
- **Pro:** Still deterministic on the consumer.
- **Con:** New header, new semantics, more to hand-edit. Cuts against the "tiny" design goal.
- **Con:** Every writer needs to think about priority or emit a default.

### D. Receiver-delegated

The consumer chooses purely on its own policy — current task type, model context window, user preference — ignoring hints in the file.

- **Pro:** Most flexible. Each consumer tunes for its own model.
- **Pro:** No protocol complexity. v1.2 ships as-is.
- **Con:** Handoffs are less predictable. A crumb emitted for Claude may load differently in Codex.
- **Con:** Pushes complexity onto every consumer equally.

### E. Hybrid (recommended shape for v1.3)

Combine B + C with D as fallback:

1. If `fold_priority=` header is present, honor writer-declared order (scheme C).
2. Otherwise, apply size-greedy with per-section fairness (scheme B): first fit `/summary` for every fold, then upgrade to `/full` in priority order until budget exhausts.
3. Consumers MAY override with their own policy (scheme D) but SHOULD document the override.

## Related decisions

- **Can a consumer load both variants?** Open. The spec doesn't forbid it, but a clean semantic is: load one or the other, never both — otherwise the fold primitive collapses into "two sections with a naming convention."
- **What about `[fold:NAME/medium]`?** Out of scope for v1.2. Revisit only if size-greedy proves insufficient.
- **Does `crumb lint` warn on oversized `/summary`?** Yes, use `--max-size` with a lower threshold for `/summary` bodies.
